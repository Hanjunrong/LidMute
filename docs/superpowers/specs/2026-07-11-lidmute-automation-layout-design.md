# LidMute 自动保护与控制区调整设计

## 目标

优化自动保护的启用条件、配置保存方式和主界面信息结构，同时保持现有合盖、模拟合盖、夜间息屏静音规则。最终 GUI 与声音行为由用户手动验收；开发过程只执行自动化测试、构建和静态检查。

## 已确认行为

### 守卫与静音

1. 单独开启守卫不修改当前音量、静音状态或媒体播放状态。
2. 真实合盖和模拟合盖会进入保护并静音内建扬声器。
3. 守卫开启、自动保护已启用、系统处于息屏状态且北京时间命中配置时段时，夜间保护仍会静音内建扬声器。
4. 如果开启守卫时系统已经处于真实合盖或有效夜间息屏状态，应立即进入对应保护。这属于当前系统状态命中，不是守卫开关无条件静音。
5. 开盖、夜间策略结束和关闭守卫继续沿用现有音量与静音恢复规则。

### 自动保护配置

1. 守卫关闭时，自动保护开关、开始时间和结束时间全部禁用并呈现置灰状态。
2. 守卫开启后，上述控件恢复可操作。
3. 删除“应用时间”按钮。
4. 自动保护开关改变后立即写入 `UserDefaults` 并重新计算保护状态。
5. 开始时间或结束时间改变且两个值均为合法 `HH:mm` 后，立即保存并重新计算保护状态。
6. 输入中的非法或未完成时间不覆盖最后一组有效配置；界面显示 `时间格式应为 HH:mm`。
7. App 关闭或重启后恢复最后一次保存的开关与有效时间。

采用“合法值立即保存”而不是依赖应用退出回调。这样即使进程被强制结束，已确认的有效配置也不会丢失。

## 界面结构

主内容区保持现有 Liquid Glass 视觉语言，但将功能重新分组：

1. `自动保护` 独立为一张玻璃卡片，包含夜间策略开关、开始时间、结束时间、当前亮屏/息屏状态和策略说明。
2. `模拟测试` 新增独立玻璃卡片，包含模拟合盖、模拟开盖和重置按钮。
3. 模拟按钮不再放在自动保护卡片底部。
4. `当前声音` 卡片继续展示进程和上一首、暂停/播放、下一首按钮。
5. 删除自动保护卡片中的“应用时间”按钮后，时间输入区域保持左右对称，不留下空白占位。
6. 所有禁用控件继续使用 Liquid Glass 按钮/输入样式，通过透明度和交互禁用共同表达状态。

布局优先保持桌面窗口中的清晰分组；窗口宽度不足时允许卡片自然纵向排列，不压缩时间字段和模拟按钮文字。

## ViewModel 数据流

- `AppViewModel` 继续从 `UserDefaults` 加载 `nightScheduleEnabled`、`nightStart` 和 `nightEnd`。
- 新增统一的自动保存入口，在开关或时间文本变化时调用。
- 时间校验继续使用现有 `minutes(from:)`；只有开始和结束都合法才写入持久化值。
- 非法输入仍保留在当前界面，便于用户修正，但不进入 `NightSchedule` 的有效配置，也不覆盖磁盘值。
- 当守卫关闭时，UI 禁用自动保护控件；ViewModel 不强制删除已经保存的自动保护设置。
- 再次开启守卫时，恢复使用已保存的自动保护开关和时间，并根据当前显示器状态立即计算是否进入夜间保护。

## 媒体控制边界

上一首、暂停/播放、下一首按钮保持现有实现和文案，继续发送 macOS 系统媒体键：

- 不增加 Chrome 网页执行权限；
- 不改用 macOS 私有 `MediaRemote` API；
- 不注入网页脚本；
- 不保证优酷或其他网页播放器响应；
- 本轮不把“点击后不生效”列为已修复结果。

保护期间自动发送的系统暂停请求同样保持上一份设计中的尽力而为语义，内建扬声器静音仍是最终保障。

## 错误与边界

- 时间输入从合法值编辑到中间态时，状态提示变为格式错误，但旧的有效设置继续保存在本机。
- 守卫关闭时修改自动保护配置不可行，因为控件被禁用；不会出现 UI 显示修改但未保存的状态。
- 自动保护配置启用但守卫关闭时，不启动夜间保护。
- 多个保护来源重叠时，继续由 `ProtectionCoordinator` 保证只要任一来源有效就保持静音。
- 媒体按钮发送成功只表示系统事件已投递，不表示目标播放器执行成功。

## 自动化测试与构建

### 行为测试

- 开启守卫但未合盖、未命中夜间息屏策略时，不静音。
- 真实合盖和模拟合盖分别静音。
- 自动保护只有在守卫开启、策略启用、息屏且命中时段时静音。
- 合法时间立即持久化；非法时间不覆盖最后有效值。
- 重启 ViewModel 配置存储后可读取最后保存的开关和时间。
- 关闭守卫后自动保护设置仍保留，但不会触发静音。

### 静态与构建验收

- `ContentView` 不再包含“应用时间”按钮。
- 自动保护控件统一绑定守卫启用状态。
- 模拟控制位于独立卡片。
- Swift 行为测试、Chrome 扩展测试、App/Native Host 构建和双次打包全部通过。
- Chrome 扩展仍不包含 `scripting` 或 `<all_urls>` 权限。

## 手动验收边界

开发代理不操作 GUI、不触发声音、不点击媒体控制，也不声称优酷已响应系统媒体键。用户自行验收：

1. 守卫关闭时自动保护控件置灰；开启后恢复。
2. 时间和开关修改后关闭并重新打开 App，配置仍存在。
3. 自动保护不再显示“应用时间”按钮。
4. 模拟合盖、模拟开盖和重置位于独立卡片。
5. 开启守卫本身不改变声音；合盖、模拟合盖或有效夜间息屏保护才静音。
6. 当前声音区域的系统媒体按钮按用户环境自行验证。
