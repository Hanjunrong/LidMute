# LidMute 保护期间系统媒体暂停设计

## 目标

守卫开启后，当真实合盖、模拟合盖或夜间息屏策略使 LidMute 进入保护状态时，在静音内建扬声器之外，尽力暂停当时正在发声的 Chrome 网页。保护结束、开盖或亮屏后不自动恢复播放。

本功能不增加网页注入权限，不读取或修改网页 DOM。它使用 macOS 系统播放/暂停媒体键，因此属于尽力而为的系统级暂停请求，不保证逐标签页暂停。

## 已确认范围

1. 触发来源包括真实合盖、模拟合盖和夜间息屏保护。
2. 仅在守卫已开启且至少一个保护来源有效时请求暂停。
3. 开盖、模拟开盖、夜间策略结束或关闭守卫时不发送播放命令。
4. 扩展继续使用现有 `tabs`、`nativeMessaging` 和 `storage` 权限，不增加 `scripting` 或 `<all_urls>`。
5. 内建扬声器静音仍是最终保障；暂停失败不得削弱或撤销静音保护。

## 方案选择

考虑过三种方案：

- 网页脚本注入：能够逐标签页调用 `HTMLMediaElement.pause()`，但需要所有网站权限，未采用。
- Chrome 标签页静音：无需读取网页内容，但媒体仍会继续播放，不满足“暂停”的目标，未采用。
- macOS 系统媒体键：无需新增网页权限，能暂停多数当前媒体会话，但目标由 macOS 决定。采用此方案，并明确其限制。

## 行为规则

### 进入保护状态

LidMute 先执行现有的内建扬声器静音流程，再判断是否需要发送媒体暂停请求。只有满足以下全部条件时才发送：

1. 守卫已开启；
2. 当前处于合盖、模拟合盖或有效夜间息屏保护；
3. CoreAudio 快照显示 Chrome 正在输出音频，或 Chrome 扩展刚上报一个 `audible: true` 的标签页；
4. 当前保护周期尚未对同一触发执行过暂停请求。

系统媒体键本质是播放/暂停切换，而不是可验证的只暂停命令。因此禁止在没有音频证据时无条件发送，避免把已经暂停的媒体重新播放。

### 保护期间再次发声

如果保护期间 Chrome 扩展上报新的标签页发声事件，且最近一次暂停请求不在去抖窗口内，则再次发送系统媒体键。请求后继续依赖扬声器静音，不根据持续的 CoreAudio 轮询高频重复发送，避免播放和暂停来回切换。

同一 Chrome `eventId` 继续由现有持久化去重器过滤。不同标签页的新事件可以触发新的暂停请求，但全局去抖窗口限制短时间内最多发送一次。初始去抖值定为 3 秒。

### 退出保护状态

退出任一保护来源时，仅按照现有规则处理扬声器状态。只要其他保护来源仍有效，就继续保持静音。所有保护来源结束后也不发送媒体键，不自动恢复网页播放。

关闭守卫时恢复现有音频快照，但不恢复任何媒体会话。

## 架构与数据流

- `ProtectionCoordinator` 继续维护守卫状态、保护来源和音频事件，不直接依赖 AppKit 或 `CGEvent`。
- Core 层新增纯决策结果，表示“有可靠 Chrome 发声证据且应请求暂停”，并携带触发来源、Chrome 进程或标签页证据以及关联 ID。
- `AppViewModel` 消费该结果，通过现有 `SystemMediaController` 发送 `.playPause`，并把发送成功或失败回报给协调器用于日志。
- `SystemMediaController` 保持系统级媒体键实现，不声称知道最终接收命令的应用。
- Chrome 扩展和 Native Messaging 协议无需新增控制网页的消息，也不增加权限。

数据流为：保护来源生效或 Chrome 发声事件 -> Core 判断是否具备音频证据及是否通过去抖 -> App 发送系统媒体键 -> Core 记录请求结果 -> 扬声器静音继续兜底。

## 日志与界面

新增两类可读事件：

- `已发送系统暂停请求`：记录真实合盖、模拟合盖、夜间息屏或保护期间 Chrome 再次发声等触发原因，以及可用的进程和标签页信息。
- `系统暂停请求失败`：记录媒体键发送错误，但不改变静音保护状态。

日志不得使用“网页已暂停”或“Chrome 已暂停”等无法验证的描述。界面状态文案使用“已请求系统暂停”或“等待媒体响应”。

现有 Chrome 标签页事件继续永久记录标题、URL、窗口 ID、标签页 ID、扩展会话 ID 和系统音频关联状态。

## 异常与边界

- macOS 可能把媒体键发送给 Chrome 之外的活跃媒体应用，这是已接受的系统级限制。
- Chrome 同时有多个发声标签页时，系统媒体键不能保证全部暂停。
- 某些网页播放器可能忽略系统媒体键；此时内建扬声器仍保持静音。
- 只有 CoreAudio 进程证据而没有 Chrome 标签页证据时，可以在进入保护状态时请求一次暂停，但日志标记为进程级证据。
- 只有 Chrome 标签页证据而 CoreAudio 尚未匹配时，也可以请求暂停，日志标记为浏览器观察证据。
- 无音频证据时不发送媒体键。
- 去抖状态在一次应用运行期间有效；应用重启后由新的保护周期重新计算。

## 测试与验收

### Core 行为测试

- 守卫关闭时，合盖、模拟合盖和夜间息屏均不产生暂停请求。
- 守卫开启但没有 Chrome 音频证据时只静音，不产生暂停请求。
- 三种保护来源分别在存在 Chrome 音频证据时产生一次暂停请求。
- 保护期间新的 Chrome 标签页发声事件在去抖窗口外产生新请求，窗口内不重复。
- 持续的相同 CoreAudio 快照不重复产生请求。
- 开盖、亮屏、夜间结束和关闭守卫均不产生播放请求。
- 媒体键发送失败会记录错误，但保护状态和扬声器静音不受影响。

### 集成验收

1. 开启守卫并播放 Chrome 视频，执行模拟合盖；内建扬声器立即静音，系统媒体暂停请求只发送一次。
2. 视频在支持系统媒体键时暂停；模拟开盖后保持暂停。
3. 启用夜间策略并模拟息屏，重复相同行为；策略结束后不自动播放。
4. 保护期间让另一个 Chrome 标签页开始发声，确认产生标签页级日志和一次受去抖约束的暂停请求。
5. 使用不响应系统媒体键的网页验证：即使网页继续播放，内建扬声器仍保持静音，界面不虚报暂停成功。
6. Chrome 扩展权限清单不新增 `scripting` 或 `<all_urls>`，现有扩展与 Native Host 连接测试继续通过。

## 非目标

- 不注入脚本调用网页 `video.pause()` 或 `audio.pause()`。
- 不保证暂停 Chrome 的每一个发声标签页。
- 不暂停或终止 Chrome 进程。
- 不在保护结束后自动恢复媒体。
- 不替换现有内建扬声器静音和恢复逻辑。
