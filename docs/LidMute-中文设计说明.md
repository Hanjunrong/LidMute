# LidMute 中文设计说明

## 一、要解决的问题

LidMute 是一个原生 macOS 菜单栏应用。开关打开后，MacBook 合盖期间只要有程序试图让**内建扬声器**输出声音，LidMute 就会立即强制静音，并把发生过程完整记录下来。开盖后恢复合盖前的音量，但内建扬声器继续保持静音。

它不干预蓝牙耳机、USB 声卡、显示器、HDMI 或其他外接输出设备。这样可以避免优酷网页、Chrome 通知、其他播放器或后台程序在合盖期间从系统外放出声，同时不会影响外接音频设备。

## 二、最终形态

应用由三部分组成：

1. `LidMute` macOS 应用
   - SwiftUI 主界面，关闭窗口后继续驻留在菜单栏。
   - 界面使用半透明材质、柔和高光、渐变光场和悬浮卡片，呈现 Liquid Glass 风格。
    - 提供总开关、实时守卫状态、内建扬声器状态、Chrome 桥接状态、活动日志和诊断测试入口。
   - 提供北京时间夜间息屏静音配置，默认 `00:00-08:00`，以及当前音频程序和系统级媒体控制按钮。

2. `LidMute Chrome Monitor` 扩展
   - 使用 Manifest V3。
   - 监听 Chrome 标签页的 `audible` 状态变化、标签页关闭和导航更新。
   - 当标签页从非发声变为发声时，立即记录该标签页，而不是只记录“Chrome 进程”。

3. `LidMuteNativeHost` 本机桥接程序
   - Chrome 扩展不能直接访问 macOS 应用，因此 Chrome 会启动这个轻量桥接程序。
   - 桥接程序通过 Chrome Native Messaging 接收标签页事件，再写入 LidMute 专用的本机收件箱；后台应用轮询该收件箱并补全日志。
   - 通信只在本机进行，不经过网络。

## 三、合盖时的行为

| 场景 | LidMute 行为 | 是否记录 |
| --- | --- | --- |
| 合盖，守卫已开启 | 保存内建扬声器原来的音量/静音值，立即静音 | 是 |
| 合盖期间任意程序开始输出 | 再次执行静音，避免其他程序改回音量 | 是，记录进程级信息 |
| 合盖期间 Chrome 标签页开始发声 | 再次执行静音 | 是，记录到标签页级信息 |
| 合盖期间切换到耳机/蓝牙/显示器 | 不静音该外接设备 | 是，记录“非内建输出，未拦截” |
| 开盖 | 恢复合盖前音量，内建扬声器继续保持静音 | 是 |
| 手动关闭守卫 | 完整恢复首次进入保护前捕获的音量与静音状态 | 是 |
| 内建扬声器不支持静音属性 | 将内建扬声器音量降为 0 | 是，明确标记“音量归零回退” |

“阻止”在第一版中的含义是：阻止声音从内建扬声器输出，而不是强行关闭网页、杀掉进程或暂停媒体。这样能保证系统稳定，也能保留足够证据追查是哪一个程序或标签页触发了播放。

## 四、息屏夜间静音

开启守卫不会改变当前正在播放的声音或音量。只有真实合盖、模拟合盖，或显示器息屏且北京时间进入配置时段时，应用才会开始静音内建扬声器。夜间时段默认是 `00:00-08:00`，界面可直接修改开始和结束时间并自动生效。时段结束时恢复进入夜间保护前的音量与静音状态；如果此时仍处于合盖保护，则继续保持静音，直到开盖或关闭守卫。

守卫关闭时，自动保护开关和时间输入框置灰且不可操作；开启守卫后恢复。自动保护开关会立即保存，两个时间都符合 `HH:mm` 后也会立即保存并自动生效，不再需要“应用时间”按钮。非法或未完成的时间文本不会覆盖最后一组有效配置，App 关闭或重启后仍恢复最后保存的设置。模拟合盖、模拟开盖和重置操作位于独立的 Liquid Glass 卡片中，不再与自动保护共用一栏。

当前音频区域展示 CoreAudio 报告的程序名、PID、Bundle ID 和可执行路径。上一首、下一首、暂停/开始按钮发送 macOS 系统级媒体按键，因此最终由当前响应系统媒体控制的播放器处理；应用不能保证后台每个程序都提供独立的进程级媒体控制接口。

保护期间如果 CoreAudio 检测到 Chrome 正在输出音频，或 Chrome 扩展上报新的发声标签页，LidMute 会发送一次 macOS 系统播放/暂停媒体键。该按键由 macOS 选择接收媒体会话，因此日志只记录“已发送系统暂停请求”，不声称网页已成功暂停。内建扬声器静音始终作为最终保障；开盖、亮屏、夜间策略结束或关闭守卫不会自动恢复网页播放。

## 五、Chrome 标签页级记录

仅靠 macOS 的 CoreAudio，最多能知道“Google Chrome 的某个进程正在输出音频”，不能知道对应的是优酷、哪个窗口或哪个标签页。Chrome 扩展补足这个缺口。

每次 Chrome 标签页在合盖保护期进入发声状态，日志会写入：

- 标签页事件时间和单调序号；
- Chrome 扩展会话 ID；
- Chrome 窗口 ID、标签页 ID、标签页索引；
- 标签页标题、完整 URL、页面加载状态；
- `audible` 由 `false` 变为 `true` 的变化；
- 标签页是否已被用户或扩展静音、静音原因；
- 是否为当前激活标签页、是否固定、是否属于无痕窗口；
- 对应的 Chrome PID、进程路径、启动时间；
- 内建扬声器名称、UID、静音/音量的拦截前后值；
- 关联结论：`已匹配系统音频`、`仅浏览器观察到` 或 `Chrome 桥接未连接`。

例如，一条可读日志可能是：

```text
2026-07-10 09:22:56.421  合盖期间检测到 Chrome 标签页发声，已强制静音内建扬声器。
标签页：优酷 - 为好内容全力以赴
URL：https://v.youku.com/v_show/id_...
Window/Tab：12/483  Chrome PID：1357
关联：已匹配系统音频  规则：built-in-speaker-muted
```

Chrome 的 `audible` 表示该标签页在最近数秒产生过声音；因此它是可靠的“标签页发声证据”，但不是网页 JavaScript `play()` 调用的逐毫秒审计。若页面尝试播放但浏览器没有实际产生音频帧，Chrome 不会把标签页标为 `audible`，系统级日志也不会虚构一次发声事件。

## 六、所需 Chrome 权限与安装边界

扩展只申请：

- `tabs`：读取标签页标题和 URL；
- `nativeMessaging`：把本机标签页事件发送给 LidMute；
- `storage`：在 Manifest V3 worker 重启后保持扩展会话 ID 和待发送事件队列。

不会申请页面注入、网页内容读取、网络请求拦截或远程上传权限。Native Messaging host 清单只允许本项目生成的扩展 ID 连接，不能使用通配符。Chrome 的官方机制要求该 host 放在用户级 `~/Library/Application Support/Google/Chrome/NativeMessagingHosts/` 目录，且路径必须指向本机可执行文件。

无痕窗口默认不在扩展监控范围内，因为 Chrome 需要用户在扩展详情页手动打开“允许在无痕模式下运行”。第一版默认不启用；如果你手动启用，LidMute 也会记录无痕标签页事件。

## 七、详细事件日志

日志永久保存于本机 Application Support 目录，采用 JSON Lines 文件，应用内展示为可筛选的时间线。每条事件都有 UUID，包含：

- 时间、序号、事件类型、规则命中原因；
- 合盖状态、守卫开关状态、模拟或真实事件标记；
- 进程名、PID、Bundle ID、可执行路径、启动时间；
- 音频设备名称、UID、传输类型、原音量、原静音值、执行后的值；
- Chrome 标签页全部字段，以及与系统 Chrome 进程的关联结果；
- 出错信息，例如设备消失、Native Messaging 断连、无可写静音属性；
- 一条可直接复制的中文摘要。

活动日志不会自动删除。系统每秒检查一次音频进程，但同一进程持续发声时只记录一次；它停止输出后再次开始，才会形成新记录。界面提供明确的“清空记录”动作，只有用户手动点击才会删除全部本机活动记录。

## 八、测试与验收

不需要频繁合盖也可验证。应用会提供“模拟合盖”和“模拟开盖”按钮，它们驱动与真实事件完全相同的保护状态机。验收包含：

1. 打开守卫，模拟合盖，确认内建扬声器被静音，外接耳机不受影响。
2. 在 Chrome 打开优酷或其他视频，确认标签页变为 `audible` 后出现 URL、标题和 Tab ID 记录。
3. 检查日志中 Chrome 标签页事件与系统 Chrome PID 的关联状态。
4. 模拟开盖，确认音量恢复为合盖前数值，但内建扬声器仍保持静音。
5. 手动关闭守卫，确认音量与静音状态完整恢复为首次进入保护前的数值。
6. 重置模拟状态，确认“模拟合盖”可点击、“模拟开盖”不可点击；点击合盖后两者状态互换。
7. 启用息屏夜间静音，确认默认 `00:00-08:00` 可修改，时段结束后恢复状态。
8. 关闭主窗口，确认菜单栏图标仍在、守卫仍可开关。
9. 断开 Chrome 扩展或 Native Messaging bridge，确认应用显示“标签页级记录不可用”，但系统级内建扬声器静音仍继续工作。
10. Chrome 正在发声时分别触发真实合盖、模拟合盖和夜间息屏保护，确认每次保护周期最多发送一次系统暂停请求。
11. 在 3 秒内连续产生多个 Chrome 发声证据，确认只记录一次系统暂停请求；3 秒后新的发声事件可产生新请求。
12. 模拟系统媒体键发送失败，确认错误被记录且内建扬声器继续保持静音。
13. 开盖、模拟开盖、亮屏、夜间时段结束和关闭守卫后，确认 LidMute 不自动恢复网页播放。

## 九、已知边界

- “标签页级”只适用于已安装 LidMute 扩展的 Google Chrome；其他浏览器保留进程级记录。
- Chrome 标签页 ID 只在该浏览器会话内唯一，日志会同时保存扩展会话 ID，避免不同会话的数字混淆。
- 记录到 Chrome 标签页发声并不等价于能够证明某段 JavaScript 调用了 `audio.play()`；本设计记录的是 Chrome 实际判定为发声的标签页，以及系统是否同时观测到 Chrome 音频输出。
- 系统媒体键只能请求当前 macOS 媒体会话切换播放或暂停，不能保证逐个暂停 Chrome 的所有发声标签页，也可能由其他活跃媒体应用接收。LidMute 不读取或修改网页内容，扬声器静音仍是最终保障。
- 部分 Mac 会将有线耳机与内建设备共用控制路径。LidMute 采用保守策略：只有 CoreAudio 当前输出数据源明确标识为 Speaker 时才静音；无法确认的共享路由会显示为不可用且不作任何修改。

## 十、技术依据

Chrome 的 `tabs.Tab.audible` 表示标签页在最近数秒内产生过声音，`tabs.onUpdated` 会在其 `audible` 状态改变时发送新值；`tabs` 权限允许读取标题与 URL。Native Messaging 允许扩展通过受 `allowed_origins` 限制的本机 host，与本机程序以带长度前缀的 JSON 消息通信。参见 [Chrome Tabs API](https://developer.chrome.com/docs/extensions/reference/api/tabs) 与 [Chrome Native Messaging](https://developer.chrome.com/docs/extensions/develop/concepts/native-messaging)。
